# Синема — смотрим фильмы вместе

Простой сайт для совместного просмотра видео: комната по коду, синхронное
воспроизведение (play/pause/перемотка) у всех участников, чат сбоку.

Источник видео — **на ваш выбор**:
- вставить прямую ссылку на видеофайл (.mp4/.webm), которое у вас есть на легальных правах;
- вставить ссылку на **YouTube** — встраивается официально через YouTube IFrame API;
- вставить ссылку на видео **VK** — экспериментально, через недокументированный
  до конца postMessage-API внешнего плеера VK. Может работать не для всех видео
  (зависит от настроек приватности видео в VK) — если не заработает, пришлите
  мне точный текст ошибки/поведения, разберёмся;
- вставить ссылку на видео **RuTube** — тоже экспериментально, через postMessage-API
  встроенного плеера RuTube. Как и с VK, точная синхронизация зависит от
  конкретного видео и его настроек — если что-то работает не так, опишите
  поведение, поправим;
- загрузить собственный видеофайл на сервер (кнопка «Загрузить свой файл»).

Никакой встроенной интеграции со сторонними «раздачами» тут нет и не будет —
такие сервисы обычно распространяют контент без прав, а GitHub блокирует
репозитории, связанные с этим, по DMCA.

## Что нового в этой версии

- **Комнаты как в Rave** — жмёшь «Создать комнату», сразу получаешь код и
  ссылку-приглашение с кнопкой «Скопировать» прямо в зале.
- **Подключение без потери места** — если человек заходит, когда видео уже
  идёт, оно не начинается заново: сервер помнит текущую секунду и
  play/pause-статус и сразу ставит нового зрителя в нужный момент.
- **YouTube, VK и RuTube ссылки** — не только файлы, см. выше.
- **Выдвижной чат стрелкой** — панель чата теперь выезжает сбоку по клику на
  стрелку-таб и одинаково работает и в обычном, и в полноэкранном режиме —
  переключать режим экрана ради чата больше не нужно. Пока панель задвинута,
  на стрелке появляется бейдж с числом непрочитанных сообщений.
- **Тонкая строка сообщений под видео** — когда чат задвинут, новое сообщение
  тонкой плашкой выезжает снизу поверх видео (имя + текст) и через пару
  секунд плавно исчезает, не мешая просмотру.
- **Новый экран входа в стиле glassmorphism** — полупрозрачная стеклянная
  карточка, плавающая иконка-хлопушка над ней, поля с плавающим лейблом и
  подсветкой при фокусе, инлайн-подсказки об ошибках вместо всплывающих
  alert-окон (с лёгкой анимацией «встряски»).
- **Адаптивная вёрстка** — от телефона до широкого монитора.
- **Современные анимации** — плавающие световые пятна на фоне, объёмное
  появление карточки входа, «дышащая» кнопка входа, плавные переходы между
  экранами, анимация сообщений в чате.

## Почему не GitHub Pages

GitHub Pages — это **только статический хостинг** (HTML/CSS/JS без сервера).
Этому сайту нужен постоянно работающий сервер (Node.js + WebSocket) для
синхронизации плеера и чата между зрителями, а также для приёма загруженных
видео. Поэтому:

- **Фронтенд** (папка `public/`) можно при желании положить на GitHub Pages.
- **Сервер** (`server.js`) нужно развернуть там, где можно запускать Node.js:
  Render, Railway, Fly.io, или собственный VPS. У всех есть бесплатные тарифы
  для старта (со своими ограничениями по времени работы и объёму диска).

Проще всего — держать и фронт, и сервер в одном месте (Render/Railway отдают
и статику, и API), не разделяя на два хостинга.

## Запуск локально

```bash
cd watch-party
npm install
npm start
```

Откройте `http://localhost:3000` в нескольких вкладках — это как разные
зрители.

## Деплой (пример на Render.com)

1. Залейте эту папку в свой репозиторий на GitHub.
2. На render.com: New → Web Service → подключите репозиторий.
3. Build Command: `npm install`
4. Start Command: `npm start`
5. После деплоя вы получите ссылку вида `https://ваш-сайт.onrender.com` —
   это и есть ссылка, которую вы даёте друзьям.

## Важно про загрузку файлов

- Файлы хранятся в папке `uploads/` прямо на сервере. На бесплатных тарифах
  диск часто **временный** — после перезапуска сервера файлы могут исчезнуть.
  Для долгоживущих видео лучше подойдёт вариант со ссылкой на файл, лежащий
  в вашем облачном хранилище (например, прямая ссылка на файл в Google Drive
  или Яндекс.Диске с открытым доступом на просмотр — если сервис отдаёт
  прямую ссылку на .mp4).
- Лимит загрузки сейчас 4GB (`server.js`, параметр `limits.fileSize`) —
  можно уменьшить под свой хостинг.
- В каждой комнате хранится один активный видеофайл (новая загрузка
  заменяет предыдущую для этой же комнаты).

## Структура проекта

```
watch-party/
├── package.json
├── server.js          # Express + Socket.io: комнаты, синхронизация, загрузка
├── public/
│   ├── index.html     # экран входа + экран зала
│   ├── style.css       # оформление
│   └── client.js       # логика синхронизации плеера и чата
└── uploads/            # сюда попадают загруженные видео (создаётся сама)
```

## Как это работает

1. Один человек создаёт зал — вводит имя и придумывает название комнаты.
2. Остальные заходят на тот же сайт и вводят то же название комнаты.
3. Любой участник вставляет ссылку на видео или загружает файл — оно
   появляется у всех.
4. Play, пауза и перемотка одного зрителя синхронизируются у всех остальных
   через WebSocket (Socket.io).
5. Рядом — простой чат для комментариев во время просмотра.

## Что было исправлено (надёжность синхронизации)

- **Главный баг рассинхрона**: защита от эхо-цикла раньше работала по
  таймеру (350мс). Если YouTube/RuTube присылали событие о смене состояния
  позже (буферизация при перемотке), это ошибочно принималось за действие
  зрителя и снова рассылалось всем — отсюда дёрганья и расхождение по
  времени. Теперь вместо таймера считается точное число ожидаемых
  эхо-событий, независимо от задержки.
- **VK был полностью нерабочим для синхронизации** — использовался
  придуманный API, которого у VK на самом деле нет. Теперь честно: видео
  показывается всем, но play/pause синхронизировать невозможно (нет
  публичного API), об этом написано прямо в интерфейсе.
- **RuTube**: был баг в названии поля состояния плеера (`status` вместо
  `state`) — из-за этого RuTube вообще никогда не считывал play/pause.
  Исправлено, синхронизация экспериментальная, но теперь действительно
  работает по документированному API RuTube.
- **Обрыв соединения** — раньше при потере связи (WiFi моргнул, телефон
  заснул) зритель навсегда "выпадал" из комнаты до перезагрузки страницы.
  Теперь клиент переподключается автоматически.
- **Блокировка автовоспроизведения браузером** — если браузер молча не
  даёт видео начать играть по команде от другого зрителя, теперь
  показывается кнопка «▶ Нажми, чтобы продолжить».
- **Ресинхронизация** — кнопка «🔄 Синхронизировать» в шапке зала и
  автоматический ресинк при возврате на вкладку (если она была свёрнута).
